## Overview

Implement automatic reconnection flow for players when network disruptions occur. This phase completes the connection health monitoring system by adding client-side awareness and recovery capabilities.

**Current State:**
- ✅ Players emit presence heartbeats every 30s
- ✅ Hosts see connection status badges
- ✅ Connection status API endpoint polling every 5s
- ✅ Player.lastSeenAt tracked in database

**What's Missing:**
- ❌ Client-side disconnection awareness
- ❌ Automatic reconnection logic
- ❌ Session state restoration after disconnect
- ❌ User feedback for disconnect/reconnect events

## Technical Decisions

### 1. Browser Network API vs Presence Failures
**Decision:** Monitor both `navigator.onLine` AND presence heartbeat failures
**Rationale:**
- `navigator.onLine` detects full network loss (WiFi off, airplane mode)
- Presence failures detect Supabase connection issues (auth expired, server down)
- Combining both gives comprehensive coverage

### 2. Reconnection Trigger Strategy
**Decision:** Auto-reconnect on `online` event, manual retry button for failures
**Pros:** User doesn't need to manually reconnect in 95% of cases
**Cons:** Could trigger unnecessary API calls if network is flaky
**Mitigation:** Debounce reconnection attempts (max 1 per 5 seconds)

### 3. State Sync Approach
**Decision:** Full session refetch via `/sync` endpoint (not delta updates)
**Pros:** Simple, guarantees consistency, handles quiz state changes during disconnect
**Cons:** More bandwidth than delta (acceptable for MVP)
**Future:** Consider WebSocket resume with delta sync for Phase 5

### 4. Offline Answer Queueing
**Decision:** **NOT in Phase 4.3** - mark as future enhancement
**Rationale:** Complex state management (queue, deduplication, reordering)
**Workaround:** Disable answer submission when offline, show "reconnect to answer"
