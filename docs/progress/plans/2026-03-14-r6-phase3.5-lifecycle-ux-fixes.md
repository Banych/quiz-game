# R6 Phase 3.5 — Game Lifecycle & UX Fixes

**Date:** 2026-03-14
**Status:** 🚧 In Progress
**Depends on:** R6 Phase 3 ✅

---

## Overview

A set of five bugs and UX gaps discovered during manual testing after Phase 3 shipped. These all touch the game lifecycle (quiz status transitions), host controls, player answer UI, and presence/connection tracking. None require new infrastructure — all are fixable within existing layers.

**Issues found:**
1. Completed quiz has no way to restart or reset to a startable state — only delete is available
2. "Resume Quiz" button on host dashboard always shows for started quizzes, but throws on press; should be conditional or replaced with contextual actions (Finish / Restart)
3. Player answer UI doesn't adapt to question type — options should show when available, free-text when not
4. No "Start Hosting" shortcut from the admin quiz detail page — host must navigate to `/host` first
5. Connected players transition to `away → disconnected` even when actively answering; status never reverts to `connected`

---

## Goals

- [ ] Quiz lifecycle: allow resetting a completed quiz back to `Draft` (or `NotStarted`) for replay
- [ ] Host dashboard: replace always-visible "Resume" with contextual action buttons based on quiz status
- [ ] Player page: render answer options that match the current question type (multiple choice vs free-text)
- [ ] Admin quiz page: add "Start Hosting" / "Open Host View" button
- [ ] Presence: fix away → disconnected regression so active players stay `connected`

## Implementation Steps

### Step 1 — Quiz Reset / Restart (Domain + Use Case + API) · Medium

**Problem:** Once a quiz reaches `Completed` status the only available action is delete. There's no path back to a startable state.

**Solution:** Add a `ResetQuizUseCase` that transitions a quiz from `Completed` (or `Active`) → `Draft`, clears `startedAt`/`endedAt`, and resets player answer state optionally.

- [x] Domain: add `reset()` method on `Quiz` entity — allowed from `Completed` or `Active` → `Pending`; throws if already `Pending`
- [x] Use case: `src/application/use-cases/reset-quiz.use-case.ts` — calls `quiz.reset()`, saves
- [x] API route: `POST /api/admin/quizzes/[quizId]/reset`
- [x] Wire into `getServices()` factory
- [x] Write domain test for `quiz.reset()` (3 tests — success from completed, success from active, throws if pending)
- [x] Write use-case test (success + not-found + invalid-state)
- [x] Run `yarn test` ✅

### Step 2 — Host Dashboard: Contextual Action Buttons · Small

**Problem:** "Resume Quiz" button renders for all `Active` quizzes and throws a 400. The host has no way to finish or restart a quiz from the dashboard.

**Solution:** Replace static "Resume" with status-aware button set:
- `Draft` → "Start" (existing)
- `Active` → "Open Live View" + "Finish Quiz"
- `Completed` → "Restart Quiz" (calls Step 1 reset endpoint) + "Delete"

- [ ] Find where "Resume Quiz" is rendered in the host dashboard component(s)
- [ ] Replace with conditional rendering based on `quiz.status`
- [ ] "Finish Quiz" button calls existing `PATCH /api/admin/quizzes/[quizId]` or dedicated endpoint
- [ ] "Restart Quiz" button calls `POST /api/admin/quizzes/[quizId]/reset` (Step 1)
- [ ] Playwright MCP spot-check: verify buttons render correctly per status
- [ ] Run `yarn test` ✅

### Step 3 — Player Answer UI: Adapt to Question Type · Medium

**Problem:** The player answer pad shows the same UI regardless of question type. For `MultipleChoice` / `TrueFalse` questions there should be option buttons; for `ShortAnswer` / `OpenText` there should be a text input.

**Solution:** Read `currentQuestion.type` and `currentQuestion.options` from the question DTO and switch the answer input component accordingly.

- [ ] Read current `PlayerSessionScreen` / answer pad component to understand existing rendering
- [ ] Add conditional: if `type === 'MultipleChoice' || type === 'TrueFalse'` → render option buttons from `question.options[]`
- [ ] Else → render text input (existing behavior)
- [ ] Ensure selected option is highlighted and submission works for both paths
- [ ] Playwright MCP spot-check: join as player, verify options appear on MC question

## Technical Decisions

**Step 1 — Reset quiz:**
Reset to `Draft` (not `NotStarted`) so the host can still edit questions before re-running. Do NOT wipe existing player answers by default — the reset is a status-only change. If answer clearing is needed, it can be a separate explicit action.

**Step 2 — Button logic:**
Keep changes in the presentation layer only. No new domain methods needed for finish/status display.

**Step 3 — Question type rendering:**
The `currentQuestion` DTO should already carry `type` and `options`. If options are missing from the DTO, extend `CurrentQuestionDTO` to include them — this is an application-layer DTO change, not a domain change.

**Step 5 — Presence fix:**
Prefer fixing the heartbeat/away threshold over patching the display layer. A display-only fix would mask the underlying issue.
