# Step 2.2: Build and Test the Application Layer

This step turns the domain contracts from Step 2.1 into orchestrated workflows that return DTOs ready for hooks and TanStack Query.

## Objectives
- Implement use cases that encapsulate discrete commands/queries (`startQuiz`, `submitAnswer`, etc.).
- Layer application services on top of those use cases so UI code can consume a single facade per feature.
- Ensure every service returns DTOs defined in `src/application/dtos/**`.
- Cover the application layer with Vitest unit tests using in-memory repositories.
- Prepare integration points (API routes, hooks) by exposing serializable results and typed errors.

## Plan
1. **Use Cases (`src/application/use-cases/`)**
   - Keep files kebab-cased (`start-quiz.use-case.ts`). Each exports a function/class with an explicit input DTO and output DTO.
   - Required flows:
     - `startQuiz`: validate quiz eligibility, transition state, emit `QuizStartEvent`.
     - `submitAnswer`: run through `Question.validateAnswer`, update scoring, broadcast `PlayerAnsweredEvent`.
     - `endQuiz`: finalize timers, compute leaderboard, emit `QuizEndedEvent`.
     - `addPlayer`, `updatePlayerStatus`, plus `findPlayerById` / `findQuizById` queries used by hooks.
   - `joinSession`: look up a quiz by join code, hydrate lobby players, and return a `QuizDTO` so the player UI can render the lobby before registration.
   - Use cases should depend only on domain entities, repositories, and value objects.

2. **Application Services (`src/application/services/`)**
   - Services group related use cases and hide orchestration details from API routes/hooks.
   - Examples:
     - `QuizService`: `start`, `end`, `getById`, `getRoundState`.
     - `PlayerService`: `join`, `updateStatus`, `listByQuiz`.
     - `AnswerService`: `submit`, `auditTrail` (future).
   - Services perform DTO mapping via helpers in `src/application/dtos/**` and surface serializable responses + error enums.
   - `src/application/services/factories.ts` centralizes Prisma repository wiring + service instantiation so API routes and tests can call `getServices()` instead of duplicating setup logic.

3. **DTO Schemas (`src/application/dtos/`)**
   - Define zod schemas for inputs/outputs. Example structure:
```ts
export const StartQuizInput = z.object({ quizId: z.string().uuid(), hostId: z.string().uuid() });
export type StartQuizInput = z.infer<typeof StartQuizInput>;
```
   - Centralize entity→DTO mappers (`quizToDTO`, `playerToDTO`). Reuse them in services, API routes, and tests.

4. **Testing (`src/tests/application/`)**
   - Build in-memory repository doubles fulfilling the domain interfaces.
   - Cover each use case + service with Vitest (`yarn test`). Use `describe("StartQuizUseCase", ...)` style for clarity.
   - For integration-style tests, wire use cases + real Prisma repo once Step 3 is ready, but define the scaffolding now (pending TODOs allowed).

5. **Error & Logging Conventions**
   - Throw typed errors (e.g., `new QuizNotFoundError(quizId)`) so API routes can map them to HTTP responses.
   - Add lightweight logger hooks (e.g., `logger.info("quiz.started", {...})`) but keep them side-effect free by default; inject logger dependencies for tests.

## Acceptance Criteria
- Every use case/service accepts typed DTO input and returns DTO output (no raw entities crossing boundaries).
- Services orchestrate repositories without leaking Prisma/HTTP specifics.
- Application tests pass via `yarn test` and cover success + failure paths.
- Documentation references (this doc + `docs/structure.md`) remain accurate—update them if new services/use cases appear.

## Next Steps
- Implement infrastructure adapters (Prisma repositories, Supabase wiring) per Step 3.
- Expose services via API routes and TanStack hooks, ensuring `yarn dev` reflects the new endpoints.
