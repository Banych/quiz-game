# Step 2.1: Define the DDD-Lite Structure

Lock in the DTO → Entity → Service → Prisma Repo → Hook flow so every future story plugs into the same architecture referenced in `docs/structure.md`.

## Objectives
- Normalize the folder layout for domain, application, infrastructure, hooks, and shared UI.
- Keep business logic framework-agnostic via entities, value objects, aggregates, and domain events.
- Document repository **interfaces** that persistence adapters must follow.
- Introduce DTO schemas (zod) that describe every payload crossing the application/presentation boundary.
- Cover the domain layer with Vitest tests before wiring real data sources.

## Tasks
1. **Folder layout**
   - Ensure the repo matches (or is a superset of) the tree below. Create missing folders; add README stubs if that aids discoverability.
```
src/
├── app/                         # Next.js routes (host, player, admin)
├── components/
│   ├── ui/                      # shadcn primitives
│   └── feature/                 # Higher-level composites
├── hooks/                       # TanStack Query + websocket hooks
├── domain/
│   ├── entities/
│   ├── value-objects/
│   ├── aggregates/
│   ├── events/
│   └── repositories/
├── application/
│   ├── dtos/
│   ├── services/
│   └── use-cases/
├── infrastructure/
│   ├── database/
│   │   ├── prisma/
│   │   └── client.ts
│   ├── repositories/
│   └── websocket/
└── lib/
```
2. **Entities & value objects**
   - Verify `Quiz`, `Question`, `Player`, `Answer`, `Timer`, `Score`, etc. enforce invariants in constructors/factories (no silent `any`).
   - Add concise comments explaining key invariants (e.g., "Quiz cannot start without at least one question").
3. **Aggregates & events**
   - Keep `QuizSessionAggregate` as the orchestration surface for quiz lifecycle state. Surface only the commands hooks will eventually call (`start`, `submitAnswer`, `end`).
   - Emit domain events (`QuizStartEvent`, `PlayerAnsweredEvent`, `QuizEndedEvent`) with DTO-ready payloads.
4. **Repository interfaces**
   - Update `src/domain/repositories/**` to expose async methods that align with the Prisma schema (Supabase Postgres). Example: `findActiveByCode(code: string): Promise<Quiz | null>`.
   - Keep interfaces slim—no Prisma types allowed here.
5. **DTO schemas**
   - Create `src/application/dtos/*.ts` with zod schemas for `QuizDTO`, `QuestionDTO`, `PlayerDTO`, `AnswerDTO`, etc.
   - Export derived types and centralize serialization logic (entities → DTOs) alongside the schemas.
6. **Domain tests**
   - Expand Vitest coverage under `src/tests/domain/**` for every entity/value object/aggregate touched above.
   - Mock clocks/randomness via `vi.useFakeTimers()` when verifying timers or scoring rules.

## Acceptance Criteria
- Folder structure matches the tree above and is documented (update `docs/structure.md` if any section moved).
- Entities/value objects cannot be instantiated in invalid states; unit tests prove the constraints.
- Repository interfaces fully describe persistence needs without leaking infrastructure details.
- DTO schemas compile and at least one service/test imports them.
- `yarn test` passes for the domain suite.

## Next Steps
Proceed to Step 2.2 to wire use cases/services that orchestrate these domain building blocks.
